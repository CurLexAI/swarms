# Linear Integration Policy

## Purpose

This policy defines the safe boundary for any future GitHub-to-Linear synchronization in this repository. It is intentionally policy-first: no automation may be enabled until the data model, secrets handling, dry-run validation, and workflow permissions satisfy this document.

## Status

- Integration state: `PLANNED_ONLY`
- External mutation: `BLOCKED_UNTIL_APPROVED`
- Default execution mode for future scripts: `DRY_RUN`

## Allowed Sync Events

Future automation may synchronize only the following event classes after human review:

| Source | Event | Allowed Direction | Notes |
| --- | --- | --- | --- |
| GitHub issue opened/edited/closed/reopened | Issue lifecycle metadata | GitHub -> Linear | Create or update a linked Linear issue summary without copying sensitive payloads. |
| GitHub issue label changed | Label metadata | GitHub -> Linear | Map only approved operational labels to Linear labels or workflow states. |
| GitHub pull request opened/edited/closed/reopened | PR lifecycle metadata | GitHub -> Linear | Link PR state to an existing Linear issue when an explicit Linear issue key is present. |
| GitHub pull request review submitted | Review status metadata | GitHub -> Linear | Sync only review state such as approved, changes requested, or commented. |
| GitHub check suite/check run completed | CI status metadata | GitHub -> Linear | Sync aggregate status only, not raw logs. |
| Linear issue status changed | Work planning metadata | Linear -> GitHub | May add a non-sensitive comment or label only when explicitly configured. |

No other event class is allowed by default. Webhook payloads must be parsed through an allowlist and rejected closed if the event type is unknown.

## Allowed Data Fields

Future synchronization may transmit only these fields:

- Repository owner/name slug.
- GitHub issue or pull request number.
- GitHub issue or pull request title after sensitive-data filtering.
- GitHub issue or pull request public URL.
- GitHub issue or pull request state.
- GitHub issue or pull request labels from an approved label allowlist.
- GitHub pull request review state.
- Aggregate CI/check conclusion: `success`, `failure`, `cancelled`, `skipped`, `timed_out`, or `neutral`.
- Linear issue key or identifier.
- Linear workflow state name from an approved state allowlist.
- Actor username or bot slug when needed for audit attribution.
- UTC event timestamp.
- Correlation ID generated by the sync service.

## Blocked Data Fields

The following data must never be synchronized to Linear, logged by sync automation, or embedded in Linear comments:

- Secrets, tokens, passwords, private keys, certificates, deploy hooks, API keys, session cookies, or bearer tokens.
- Raw endpoint URLs for private Modal, Render, MCP, GitHub webhook, internal control-plane, mTLS, MDM, eSIM, private 5G, device-management, or customer environments.
- `.env` contents or secret-store values.
- Customer names, customer identifiers, contractual identifiers, billing data, tenant IDs, or case-specific legal data.
- Device identifiers, serial numbers, IMEI, ICCID, EID, UDID, MAC addresses, IP addresses, TPM attestation evidence, MDM enrollment payloads, posture evidence, or telemetry tied to a device or user.
- SAMA, PDPL, NCA, incident-response, vulnerability, or audit evidence that is not already approved for external task tracking.
- Raw CI logs, stack traces containing environment data, diffs, source archives, screenshots, database records, or file contents.
- AI prompts, model outputs, or agent traces unless separately reviewed and redacted.

If a payload contains blocked data, the sync must fail closed and record only a local redacted audit event.

## Secrets and Configuration

- Linear credentials must be read from GitHub Secrets or an approved runtime secret store only.
- `LINEAR_API_KEY` must never be committed with a real value.
- Repository variables are not an acceptable substitute for secrets.
- Workflow logs must never echo secret values, endpoint values, webhook payload bodies, or HTTP authorization headers.
- Placeholder names may exist in `.env.example` solely to document required configuration.

## GitHub Actions Requirements

If a workflow is added later, it must meet all of these requirements:

1. Use minimal permissions, starting with `contents: read` and adding only the specific permissions required for the event path.
2. Read `LINEAR_API_KEY` from `${{ secrets.LINEAR_API_KEY }}` only.
3. Avoid `pull_request_target` unless a separate security review approves the exact threat model.
4. Never run untrusted pull request code with Linear credentials available.
5. Mask any configured secret values before validation steps.
6. Default to dry-run on pull requests and require explicit manual approval for mutation.
7. Upload only redacted local audit summaries when artifacts are needed.

## Dry-Run Requirement

Any Linear synchronization script must implement a non-mutating dry-run mode before it can be used in CI or by operators.

Minimum dry-run behavior:

- Accept an explicit flag such as `--dry-run` or environment variable such as `LINEAR_SYNC_DRY_RUN=true`.
- Make no external mutation requests to Linear or GitHub.
- Print a redacted action plan showing the event type, target object key, allowed fields, and blocked fields count.
- Return a non-zero exit code if blocked data is detected.
- Be covered by tests or a documented command that validates parsing and redaction without network access.

Mutation mode must remain disabled until a maintainer reviews the dry-run output and approves the exact event mapping.

## Validation Checklist for Future Automation

Before enabling automation, verify:

- The event allowlist is implemented and tested.
- The field allowlist is implemented and tested.
- Blocked data patterns fail closed.
- Dry-run mode works without secrets and without network mutation.
- GitHub Actions permissions are minimal.
- `LINEAR_API_KEY` is present only in GitHub Secrets or another approved secret store.
- No private endpoint URL, customer data, or device data appears in logs or Linear payload previews.

## Design Decision

The integration is constrained as a policy and redaction boundary first, not as an active connector. This keeps the repository in a safe planned-only state while allowing future automation to be reviewed against explicit event, field, secret, and dry-run requirements.

## Layer Impact

- Device Layer: no device identifiers, posture evidence, or MDM/eSIM data may sync.
- Control Layer: no internal endpoint URLs, mTLS material, or policy payloads may sync.
- Connectivity Layer: no carrier, eSIM, private 5G, or network identifiers may sync.
- Decision Layer: no AI prompts, model outputs, agent traces, or private runtime endpoints may sync without separate review and redaction.
